# v139 — стабилизационный релиз от v138

## Цель
Убрать архитектурные «детские болезни», которые тянулись через версии: ранний запуск `init()`, конфликтующие обработчики кнопки динамики метрик, тихие битые пути, рассинхрон версии приложения и manifest, а также устаревшие ID элементов легенды.

## Изменения

1. `init()` больше не запускается в середине `app.js`. Запуск перенесён в финальный bootstrap в самом конце файла, после установки всех wrappers/patches.
2. Кнопка `openTopologyTrends` нормализуется после инициализации: старые обработчики удаляются через замену DOM-узла, остаётся один финальный обработчик.
3. Исправлена ссылка `data/hydro/north_cap.geojson` → `data/reference/north_ocean_cap.geojson`.
4. Устаревшие ID `hydroToggle`, `railToggle`, `centersToggle` заменены на фактические `toggleHydro`, `toggleRailways`, `toggleCenters`.
5. `APP_VERSION`, query-string в `index.html` и `data/manifest.json` синхронизированы на v139.
6. Фильтр рёбер потенциальной связности получил числовые поля для точного ввода min strength и max impedance, плюс расширенные пороги impedance до 5000.
7. Добавлен boot-error overlay: если Leaflet не загрузился, пользователь видит понятное сообщение вместо молчаливого падения `L is not defined`.

## Рекомендуемый пайплайн дальнейшей разработки

### 0. Freeze / baseline
Перед любой новой функцией фиксировать рабочий baseline: ZIP, patch, manifest, список smoke-тестов. Новая версия не должна начинаться с «поверх старого хаоса».

### 1. Инвентаризация ядра
Разделить код на зоны ответственности: загрузка данных, карта, слои, фильтры, аналитика, экспорт, UI widgets. Временные патчи `vXX` не должны становиться постоянной архитектурой.

### 2. Bootstrap-first
Любая версия должна иметь один вход: `bootstrap()` → `installPatches()` → `init()` → `postInitSmokeChecks()`. Вызов `init()` в середине файла запрещён.

### 3. Contract для DOM ID
Все ID элементов фиксируются в одном словаре/registry. Код не должен обращаться к `hydroToggle`, если в HTML есть только `toggleHydro`.

### 4. Tool ownership
У каждого инструмента один владелец: один init, один bind, один render, один cleanup. Для панели графиков, экспорта, легенды и графа связности нельзя оставлять конкурирующие обработчики.

### 5. Data manifest as source of truth
Все пути к данным должны идти через `manifest.json`, кроме явно документированных статических reference-слоёв. При сборке проверять, что каждый путь существует.

### 6. Smoke tests перед ZIP
Минимум: JS syntax, JSON parse, manifest path check, version sync, отсутствие раннего `init()`, отсутствие известных legacy ID, проверка ключевых DOM ID.

### 7. Feature branches
Новые инструменты развивать отдельными блоками: `feature/export`, `feature/trends`, `feature/connectivity`, `feature/boundary-overlays`. Сначала стабилизация API, потом визуальные улучшения.

### 8. Release notes
Каждая версия получает краткий README: что менялось, какие риски, какие тесты пройдены. Это особенно важно для атласа, где данные и интерфейс развиваются одновременно.
